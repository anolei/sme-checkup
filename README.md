# 优质中小企业自检助手（开源版）

[![License: MIT](https://img.shields.io/badge/License-MIT-blue.svg)](https://opensource.org/licenses/MIT)
[![Version](https://img.shields.io/badge/version-v1.0--open-green)]()
[![指标版本](https://img.shields.io/badge/指标集-小巨人对标v1.0-orange)]()

一款**开箱即用、完全离线处理、无外部网络 API 依赖**的中小企业全方位"健康体检"AI 技能（Skill）。

通过快速、简单地调用本技能，即可对中小企业进行**从财务健康度到知识产权壁垒，从产业链定位到核心技术叙事**的全方位自检。AI 辅助可自动穿透验算数据，精准定位企业隐性风险。

其核心作用是：作为一套严格的规则化审查与诊断流程，结合结构化规则、文本解析与财务数据复核逻辑，对材料进行全流程、多维度体检，帮助企业自主查漏补缺，提升材料的完整性、一致性与可读性。

> [!NOTE]
> **当前指标版本：v1.0（小巨人对标指标集）**
> 本初版以国家级专精特新"小巨人"企业的评价指标作为对标基准。本技能采用可插拔的指标体系架构，后续可无损扩展接入省级专精特新、创新型中小企业等更多层级的对标指标集。

> [!WARNING]
> 本工具输出的任何体检报告，均基于人工智能技术结合输入文本与规则库生成，主要用于逻辑互查、规则排雷与文本优化辅助。由于模型无法调阅底层财务账册、审计底稿及企业核心机密信息，本报告不可替代正式审计、专业中介机构意见或主管部门最终评审结论。

## 🌟 核心特性

- 🕵️ **合规红线预检**：自动提取并计算营业收入、主营业务占比、两年度营收增长率、研发费率等关键底线指标，快速对照政策条款进行预判。
- 💸 **财务数据穿透式验算**：自动校验申报金额、增长率与关键财务口径之间的一致性，识别常见的口径不统一、前后不自洽等问题。
- ⛓️ **跨表单逻辑扫描**：不仅检查单项填报，还会交叉比对行业代码、领域归属、主导产品、主营业务占比等关键字段的一致性。
- 📖 **重点叙事板块诊断**：围绕产业链配套说明、产学研合作、企业简介、主导产品名称等关键主观填报板块，提供问题识别与优化建议。
- 📄 **双格式报告输出**：支持生成 Markdown 与 PDF 两种格式的诊断报告，便于内部审阅、沟通与归档。
- 🛡️ **自检闸门机制**：内置多项退出校验逻辑，未完整执行核心诊断流程时可触发回炉复检，尽量减少遗漏与敷衍输出。

## 🏗️ 可扩展架构

本技能采用**指标体系与诊断引擎分离**的可插拔架构设计，确保了**零侵入式升级**能力：

```
┌─────────────────────────────────────────────────────┐
│                   诊断引擎层                          │
│            SKILL.md 六阶段工作流（通常无需修改）        │
├──────────┬──────────┬──────────┬─────────────────────┤
│ 指标基准层 │ 知识规则层 │ 产业图谱层 │    报告模板层       │
│ review_  │ expert_  │ key_ind- │    report_          │
│ criteria │ knowledge│ ustry_   │    template.md      │
│ .json    │ .md      │ chain_   │                     │
│ (可替换)  │ (可追加)  │ map.json │    (可定制)         │
│          │          │ (可扩展)  │                     │
└──────────┴──────────┴──────────┴─────────────────────┘
```

**扩展方式**：
- 接入新指标体系 → 仅需新建/替换 `review_criteria.json`
- 新增行业经验规则 → 在 `expert_knowledge.md` 末尾追加编号条目
- 扩展产业链覆盖 → 在 `key_industry_chain_map.json` 中增减条目

## 🚀 部署与使用指南

### 一、依赖安装

本技能内置基于 Python 的自动渲染工具链，使用 Playwright 进行 PDF 渲染，跨平台部署相对简洁。

仅需在终端执行：

```bash
pip install -r requirements.txt
playwright install chromium
```

### 二、推荐使用的平台

本项目以 `SKILL.md` 为核心入口，建议优先放在支持 Skills、自动化任务、本地工具调用和文件读写的平台中使用。当前较适合本类开源 Skill 的平台，推荐按以下顺序列出：

* **OpenClaw**
* **钉钉悟空**
* **CoPaw**
* **腾讯 WorkBuddy**
* **腾讯 CodeBuddy**

> 除上述平台外，其他原生支持 Skills 机制的工具与平台，理论上亦可适配本项目。  
> 本项目的核心入口为 `SKILL.md`，凡支持类似技能加载、系统指令注入、工具调用及文件读写能力的平台，通常均可迁移接入。  
> 但需要注意的是，不同工具的平台能力、执行权限、上下文管理方式、文件处理链路以及所接入大模型的能力差异较大，最终输出效果可能存在明显波动。  
> 因此，对于诊断结论、规则判断和改写建议，使用者应结合具体平台特性与实际生成结果，保持必要的审慎识别，并进行人工复核与校正。

### 三、开发调试环境

以下工具适合用于本地调试、脚本开发、仓库维护和效果验证：

* **Cline**
* **OpenHands**
* **Cursor**
* **Trae**

### 四、接入方式说明

本仓库的核心运转逻辑封装在 `SKILL.md` 中。实际接入时，通常可采用以下方式：

* 将 `SKILL.md` 作为系统提示词、技能主指令或核心工作流入口
* 将 `references/` 作为规则基准库或知识库内容
* 将 `scripts/` 作为可调用脚本工具
* 将 `reports/` 作为最终报告输出目录

在支持 Skills 的平台中，可直接以 Skill 方式调用；  
在不支持原生 Skills 的平台中，也可通过"系统提示词 + 工具调用 + 文件读写"的方式进行迁移接入。

**典型调用示例：**

* **"按照这个 skill，帮我体检一下这份材料。"**
* **"读取这份材料，按技能流程输出诊断报告。"**
* **"先抽取文本，再执行合规检查和财务验算，最后生成 Markdown 和 PDF 报告。"**

AI 即会根据该技能内的规则与工作流，自动执行材料解析、逻辑诊断、规则核验和报告输出。

## 📂 项目架构

```text
sme-checkup/
├── SKILL.md                       # 技能主指令入口（核心工作流引擎）
├── README.md                      # 本说明文件
├── LICENSE                        # MIT 开源协议许可
├── requirements.txt               # Python 及渲染依赖项
├── self_review.md                 # 核心审查退出清单定义
├── reports/                       # 生成的诊断报告输出目录
├── scripts/                       # 引擎控制脚本
│   ├── extract_text.py            # 文档核心内容提取器
│   ├── self_review.py             # 自检闸门复核脚本
│   └── generate_diagnosis_pdf.py  # PDF 渲染生成脚本
├── references/                    # 规则与知识基准库（可插拔替换）
│   ├── review_criteria.json       # 【指标基准层】核心量化阈值（当前：小巨人 v1.0）
│   ├── key_industry_chain_map.json # 【产业图谱层】重点产业链映射参考
│   ├── expert_knowledge.md        # 【知识规则层】典型问题、红线定义与优化参考
│   └── report_template.md         # 【报告模板层】标准报告模板
└── skills/
    └── expert-dual-assessment/    # 内嵌子模块：双维度产业赛道分析引擎
```

## 📌 使用边界说明

本工具的定位是**企业材料自查辅助工具**，而非"包过工具"或替代正式评审的自动化系统。  
对于模型生成的诊断结论、修改建议和风险提示，使用者仍应结合企业真实情况、最新政策要求及专业人员意见进行人工判断，不宜脱离实际直接照单采用。
